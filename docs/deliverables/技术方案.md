# EduAgent — 可嵌入式跨课程 AI Agent 通用架构平台

## 详细技术方案

**赛题**: 第十七届中国大学生服务外包创新创业大赛 A25  
**出题方**: 中国计量大学  
**日期**: 2026-04-06

---

## 1. 项目概述

### 1.1 背景与动机

当前高校教育信息化建设中，AI 辅助教学系统面临三大核心痛点：

1. **碎片化重复建设**：各课程独立开发 AI 系统，高等数学、大学物理等课程各自为政，导致开发资源浪费；
2. **数据孤岛**：学生在不同课程中的学习数据相互割裂，无法进行跨课程关联分析和知识迁移；
3. **平台割裂**：教师和学生需要在超星、钉钉等多个平台间反复切换，体验碎片化。

EduAgent 旨在通过构建一套统一的 AI Agent 通用架构平台，从根本上解决上述问题。

### 1.2 核心目标

构建一套支持多课程 AI Agent 快速开发的通用平台架构，可嵌入主流教学平台，实现跨课程知识迁移与学情分析，提供位置级精细化作业批注。

### 1.3 硬性指标

| 指标 | 要求 | 实现策略 |
|------|------|---------|
| 并发用户 | ≥500 | FastAPI async + Redis 缓存 + Nginx 限流 |
| 知识点回答正确率 | ≥95% | RAG 二阶段检索 + Neo4j 图谱增强 |
| 作业批注准确率 | ≥95% | GPT-5.4 JSON Mode + Few-shot + 段落预编号 + 校验降级 |
| 平台兼容 | ≥2 个教学平台 | 超星（LTI 1.3）+ 钉钉（开放平台 SDK） |

### 1.4 演示课程

- 高等数学 + 大学物理
- 跨课程知识迁移示例：微积分 → 牛顿力学（做功定理）

## 3. M1: 统一 AI Agent 框架

### 3.1 LangGraph Director Graph 编排模式

核心采用 Director Graph 编排模式（移植自清华 OpenMAIC），由一个 Director Agent 作为中央路由器，根据用户意图动态分发至 5 个专业 Agent。

**状态机流转**：

```
START → director → agent_generate → director (loop) → END
```

Director 在每轮对话中分析用户意图，选择最合适的 Agent 执行任务。若 Agent 返回结果需要进一步处理，Director 可将结果传递给另一个 Agent，形成多 Agent 协作链。

### 3.2 Agent SDK

提供声明式 Agent 注册 SDK，新增 Agent 仅需继承 BaseAgent 并注册：

```python
@registry.register("qa")
class QAAgent(BaseAgent):
    name = "智能答疑 Agent"
    description = "回答课程知识问题，基于 RAG 检索知识库"
    
    async def handle(self, message, ctx) -> AsyncGenerator:
        docs = await ctx.knowledge.search(message, ctx.course_id)
        async for chunk in ctx.llm.stream(prompt, docs):
            yield TextDelta(content=chunk)
```

### 3.3 五个专业 Agent

| Agent | 职责 | 触发意图 | 调用服务 |
|-------|------|---------|---------|
| **QA Agent** | 智能答疑 | 知识问答类问题 | KnowledgeService（RAG + 图谱检索） |
| **Grader Agent** | 作业批改 | 提交作业/请求批改 | GradingService（位置级批注引擎） |
| **Tutor Agent** | 练习生成 | 请求练习/巩固 | AnalyticsService（BKT 选题） |
| **Analyst Agent** | 学情分析 | 查看成绩/能力画像 | AnalyticsService（画像 + 预警） |
| **Meta Agent** | 课程配置 | 配置 Agent/知识库 | AgentRegistry + Config |

### 3.4 SSE 流式协议

采用 AG-UI 兼容的 SSE 事件协议，支持前端实时渲染：

```
agent_start → text_delta* → action* → agent_end
```

事件格式：
```json
{
  "event": "text_delta",
  "data": {
    "agent": "qa",
    "content": "根据知识库资料...",
    "sources": [{"doc_id": "...", "chunk": "..."}]
  }
}
```

## 4. M2: 知识中间件

### 4.1 三层知识架构

#### 第一层：文档 RAG（LangChain + Milvus）

| 环节 | 技术 | 参数 |
|------|------|------|
| 文档解析 | Unstructured | 支持 Word/PDF/PPT/LaTeX |
| 文本切片 | RecursiveCharacterTextSplitter | chunk_size=1000, overlap=200 |
| 向量化 | text-embedding-3-large | 3072 维 |
| 索引 | Milvus IVF_FLAT | Top-10 候选 |
| 精排 | GPT-5.4 Rerank | Top-10 → Top-3 |

**检索流程**：
```
用户问题 → Embedding → Milvus 语义检索 Top-10 
    → GPT-5.4 Rerank 重排序 → Top-3 最相关片段
    → 拼接 Context → LLM 生成回答
```

#### 第二层：知识图谱（Neo4j）

知识图谱使用 Neo4j 存储课程知识点间的结构化关系：

- **节点类型**：KnowledgePoint（id, name, course, difficulty, embedding）
- **边类型**：
  - `PREREQUISITE`：前置依赖关系
  - `CONTAINS`：包含关系
  - `CROSS_COURSE`：跨课程关联
  - `SIMILAR_TO`：相似知识点

跨课程迁移示例：
```cypher
MATCH path = (a:KnowledgePoint {id: "MATH-CALC-001"})
  -[:CROSS_COURSE]->(b:KnowledgePoint {course: "physics"})
RETURN path
// 微积分 → 牛顿力学（做功定理）
```

#### 第三层：标准化接口

统一知识点 JSON Schema 表示，实现跨 Agent 知识共享：

```json
{
  "id": "MATH-CALC-001",
  "name": "定积分基础",
  "course": "高等数学",
  "difficulty": 3,
  "prerequisites": ["MATH-CALC-000"],
  "cross_course": ["PHY-MECH-003"]
}
```

### 4.2 知识中间件 API

| 端点 | 方法 | 功能 |
|------|------|------|
| `/knowledge/upload` | POST | 上传课程文档，触发解析和索引 |
| `/knowledge/search` | POST | 语义检索 + 重排序 |
| `/knowledge/graph` | GET | 获取知识图谱子图 |
| `/knowledge/points` | GET | 获取课程知识点列表 |

| `/knowledge/cross-course` | GET | 跨课程关联推荐 |

## 5. M3: 平台适配层

### 5.1 超星 LTI 1.3 对接

使用 ltijs（Node.js 开源库）实现 LTI 1.3 Provider，独立 Docker 容器运行（端口 3000）。

**对接流程**：

```
超星 → OIDC Login Init → ltijs → ID Token 验证
  → 提取身份信息(user_id, role, course)
  → REST 转发到 FastAPI → 签发 JWT
  → 302 重定向至 Next.js 前端（携带 JWT）
```

**成绩回传**：
```
Grader Agent 批改完成 → GradingService 生成分数
  → REST 调用 ltijs → lti.Grade.submitScore()
  → 成绩写回超星教务系统
```

### 5.2 钉钉开放平台集成

三种集成模式：

| 模式 | 功能 | 实现 |
|------|------|------|
| 企业内部机器人 | 接收消息 → Agent 编排 → 回复 | Webhook + FastAPI 路由 |
| 工作通知 | 学情预警推送 | 钉钉开放 API |
| H5 微应用 | 嵌入学生端完整界面 | iframe 嵌入 Next.js |

### 5.3 统一身份层

`platform_users` 映射表，将不同平台的用户 ID 统一为系统内部用户：

```sql
CREATE TABLE platform_users (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    platform VARCHAR(20),        -- 'lti' / 'dingtalk'
    platform_user_id VARCHAR(255),
    metadata JSONB,
    UNIQUE(platform, platform_user_id)
);
```

解析优先级：JWT Token → LTI user_id → 钉钉 union_id → 本地注册。

## 7. M5: 精细化智能引擎

### 7.1 位置级批注 Pipeline（4 阶段）

#### 阶段 1：文档预处理

将学生作业按段落编号，生成唯一锚点 ID：
```
[P1] 首先对函数 f(x) = x² 求导...
[P2] 由牛顿-莱布尼茨公式可得...
[P3] 计算结果为 1/2...
```

#### 阶段 2：GPT-5.4 结构化批改

使用 JSON Mode + Few-shot（5 个示例）输出结构化批注：

```json
{
  "annotations": [
    {
      "paragraph_id": "P4",
      "char_start": 12,
      "char_end": 15,
      "original_text": "1/2",
      "type": "error",
      "severity": "critical",
      "comment": "计算错误：1/3 ≠ 1/2",
      "correction": "1/3",
      "knowledge_point": "MATH-CALC-001"
    }
  ],
  "overall_score": 75,
  "summary": "整体思路正确，定积分公式应用有误..."
}
```

#### 阶段 3：规则引擎校验

对 LLM 输出进行四重校验：
1. `paragraph_id` 存在性验证
2. `char_start`/`char_end` 范围合法性检查
3. `original_text` 与原文精确匹配
4. `knowledge_point` 在知识库中存在

不通过的批注降级为段落级批注或移除。

#### 阶段 4：知识点关联

批注中的 `knowledge_point` 触发后续流程：
```
批注错误 → 更新 BKT 掌握度 → 推荐相关练习 → 形成"测-评-练"闭环
```

### 7.2 准确率保障策略

| 策略 | 累计准确率 |
|------|-----------|
| GPT-5.4 JSON Mode（结构化输出） | ~85% |
| + Few-shot 示例（5 个标注样本） | ~90% |
| + 段落预编号（消除位置歧义） | ~93% |
| + original_text 精确匹配校验 | ~96% |
| + 低置信降级（丢弃不确定批注） | ~97%+ |

### 7.3 BKT 知识追踪算法

移植自 UC Berkeley OATutor 项目的贝叶斯知识追踪算法（14 行核心代码）：

```python
def bkt_update(params, is_correct):
    """贝叶斯知识追踪更新"""
    if is_correct:
        num = params['pM'] * (1 - params['pS'])
        den = (1 - params['pM']) * params['pG']
    else:
        num = params['pM'] * params['pS']
        den = (1 - params['pM']) * (1 - params['pG'])
    posterior = num / (num + den)
    params['pM'] = posterior + (1 - posterior) * params['pT']
```

**四个参数含义**：
- `pM`（Mastery）：当前掌握概率
- `pT`（Transition）：从未掌握到掌握的转移概率
- `pG`（Guess）：未掌握但答对的猜测概率
- `pS`（Slip）：已掌握但答错的失误概率

### 7.4 选题策略

```python
def select_problem(bkt_states, problem_pool):
    """选择掌握度最低的知识点对应题目"""
    candidates = [
        (kp, state) for kp, state in bkt_states.items()
        if state['pM'] < 0.95  # 已掌握则跳过
    ]
    candidates.sort(key=lambda x: x[1]['pM'])
    target_kp = candidates[0][0]
    return random.choice(problem_pool[target_kp])
```

### 7.5 "测-评-练"闭环

```
作业提交 → Grader Agent 批改定位错误知识点
    → BKT 更新掌握度
    → Tutor Agent 依据掌握度最低优先选题
    → 学生完成练习（题库 / LLM 动态生成）
    → 更新 BKT 掌握度
    → 循环直至 pM ≥ 0.95
```

| 异步 I/O | FastAPI 全链路 async/await |
| 连接池 | SQLAlchemy AsyncSession + asyncpg |
| 缓存 | Redis 缓存热点知识点、会话、BKT 状态 |
| 限流 | Redis 令牌桶算法，按用户/IP 限流 |
| LLM 排队 | asyncio.Semaphore 限制并发 LLM 调用 |
| 静态资源 | Nginx gzip + CDN 加速 |
| 数据库索引 | PostgreSQL B-tree + GIN（JSONB） |

### 9.3 服务依赖关系

```
nginx → frontend → backend → postgres, neo4j, milvus, redis
                  → lti-provider → postgres
```

## 10. 创新点

### 10.1 七大核心创新

| # | 创新点 | 描述 |
|---|--------|------|
| 1 | **Director Graph 多 Agent 编排** | 基于 LangGraph 状态图的 Director 模式，1 个路由 + 5 个专业 Agent 协作，支持多轮对话和 Agent 间任务传递 |
| 2 | **位置级精细化批注** | 业界首创段落+字符级定位的 AI 批注，通过 4 阶段 Pipeline 实现 ≥95% 准确率 |
| 3 | **跨课程知识迁移** | Neo4j 图谱 CROSS_COURSE 边实现课程间知识关联，RAG 检索自动引入相关课程资料 |
| 4 | **BKT 自适应练习** | 移植 OATutor 贝叶斯知识追踪，依据个体掌握度动态选题，实现个性化学习路径 |
| 5 | **三位一体嵌入** | CopilotKit Popup（超星 iframe）+ CopilotKit Sidebar（钉钉 H5）+ Chrome 扩展（任意平台），一套代码三种形态 |
| 6 | **测-评-练闭环** | 批改定位 → BKT 追踪 → 智能选题 → 练习巩固 → 掌握度更新，完全自动化的学习闭环 |
| 7 | **零代码 Agent 构建** | Meta Agent 提供可视化配置界面，教师无需编程即可配置 Agent 的模型、Prompt 模板和知识库 |

### 10.2 与现有方案的差异

| 维度 | 传统方案 | EduAgent |
|------|---------|----------|
| Agent 架构 | 单一 LLM 调用 | Director Graph 多 Agent 编排 |
| 批注粒度 | 整体评语 | 段落+字符级精确定位 |
| 知识关联 | 课程内孤立 | Neo4j 跨课程知识图谱 |
| 练习推荐 | 固定题库顺序 | BKT 个性化选题 |
| 平台兼容 | 独立部署 | LTI 1.3 + 钉钉 SDK 嵌入 |
